## 🔄 Binary Search Pattern Modification for Doubles

### Standard Integer Binary Search Pattern

```python
# Standard pattern used in ALL previous binary search problems
def binary_search_integer(arr, target):
    low = 0
    high = len(arr) - 1
    
    while low <= high:  # Standard condition
        mid = (low + high) // 2
        
        if check(mid):
            # Found potential answer
            ans = mid
            high = mid - 1  # Eliminate right half
        else:
            low = mid + 1   # Eliminate left half
    
    return ans
```

**Key characteristics:**
- Condition: `low <= high`
- Updates: `low = mid + 1` and `high = mid - 1`
- Works perfectly for integers

---

### 🚫 Why Standard Pattern Fails for Doubles

> **Critical Understanding**: When the answer is a `double` or `long double`, the standard pattern breaks down completely.

#### Problem 1: Cannot Use `mid + 1` or `mid - 1`

```python
# ❌ WRONG for doubles
mid = 5.234567
low = mid + 1  # What does +1 mean for a double?
               # Jumps from 5.234567 to 6.234567
               # Skips ALL values between them!

# ❌ Also WRONG
low = mid + 0.1      # Still skips 5.234568, 5.234569, etc.
low = mid + 0.0001   # Still skips many values
low = mid + 0.000001 # Better but still problematic
```

**Why this fails:**
- Doubles have infinite precision (theoretically)
- Any fixed increment skips infinite values
- Answer might lie in the skipped range

#### Problem 2: Loop Termination Issues

```python
# ❌ WRONG for doubles
while low <= high:  # When do doubles become "equal"?
    # 5.2345678 vs 5.2345679 - are these equal?
    # Computer precision makes this unreliable
```

**Issues:**
- Floating-point comparison is imprecise
- May loop infinitely or terminate prematurely
- Different machines may behave differently

---

### ✅ Modified Pattern for Doubles

```python
def binary_search_double(low, high, precision=1e-6):
    """
    Binary search for problems with double/float answers
    
    Args:
        low: Lower bound of search space
        high: Upper bound of search space
        precision: Required decimal precision (default: 6 decimal places)
    
    Returns:
        Answer as a double
    """
    # Key Difference 1: Different loop condition
    while (high - low) > precision:  # Not low <= high
        mid = (low + high) / 2.0  # Use float division
        
        if check(mid):
            # Key Difference 2: No +1 or -1
            high = mid  # Not mid - 1
        else:
            low = mid   # Not mid + 1
    
    # Either low or high works as answer
    return high  # or low, or (low + high) / 2
```

---

### 🎯 Understanding the New Loop Condition

#### The Precision-Based Termination

```python
while (high - low) > 1e-6:
```

**What this means:**
- Continue until `low` and `high` are within `0.000001` of each other
- First 6 decimal places must match
- Example: `5.234567123` and `5.234567456` → STOP (first 6 decimals match)

#### Visual Example

```
Iteration 1: low = 0.0,      high = 1.0       → diff = 1.0      > 1e-6 ✓ Continue
Iteration 2: low = 0.0,      high = 0.5       → diff = 0.5      > 1e-6 ✓ Continue
Iteration 3: low = 0.25,     high = 0.5       → diff = 0.25     > 1e-6 ✓ Continue
Iteration 4: low = 0.375,    high = 0.5       → diff = 0.125    > 1e-6 ✓ Continue
...
Iteration N: low = 0.499999, high = 0.500000  → diff = 0.000001 ≤ 1e-6 ✗ STOP
```

#### Precision Mapping

| Problem Requirement | Loop Condition | Precision |
|---------------------|----------------|-----------|
| 6 decimal places | `(high - low) > 1e-6` | `0.000001` |
| 7 decimal places | `(high - low) > 1e-7` | `0.0000001` |
| 4 decimal places | `(high - low) > 1e-4` | `0.0001` |
| 10 decimal places | `(high - low) > 1e-10` | `0.0000000001` |

### 🔍 Understanding `mid` (No Increment/Decrement)

#### Why We Use Exact `mid`

```python
# Standard integer approach
if check(mid):
    high = mid - 1  # ❌ For doubles, this skips infinite values

# Double approach
if check(mid):
    high = mid      # ✅ Keeps mid as potential answer
```

**Reasoning:**

```
Search space: [0.0, 1.0]
mid = 0.5

If 0.5 is possible:
    ❌ high = 0.5 - 1 = -0.5  (nonsense)
    ❌ high = 0.5 - 0.1 = 0.4 (skips 0.41, 0.42, ..., 0.49)
    ✅ high = 0.5             (keeps 0.5 as upper bound)
    
Next iteration will check mid = 0.25
Then mid = 0.125, etc.
Converges to optimal answer
```

        else:

# ═══════════════════════════════════════════════════════════
def binary_search_double():
    low = 0.0
    high = max_value
    
    while (high - low) > 1e-6:      # Precision-based

```

| Aspect | Integer Pattern | Double Pattern |
|--------|----------------|----------------|
| **Loop Condition** | `low <= high` | `(high - low) > precision` |
| **Mid Calculation** | `(low + high) // 2` | `(low + high) / 2.0` |
| **Update on Success** | `high = mid - 1` | `high = mid` |
| **Update on Failure** | `low = mid + 1` | `low = mid` |
| **Answer Storage** | Store in variable | Return `high` or `low` |
| **Termination** | When `low > high` | When difference < precision |

## 🎯 Problem-Specific Binary Search Implementation

### Understanding the Search Space

#### What Are We Searching For?

> **Answer**: The minimum possible value of the maximum distance between consecutive gas stations.

Let's break this down with an example:

```
Gas stations: [1, 2, 3, 4, 5, 12]
k = 4 (new stations to place)

Current max distance = 12 - 5 = 7 (between positions 5 and 12)

Question: What's the MINIMUM we can make this MAXIMUM distance?
```

#### Why Binary Search on Answer?

```python
# Possible answers form a continuous range
# We need to find the MINIMUM answer that's POSSIBLE

Answer = 0.0:   Can we achieve max distance of 0.0? → Need infinite stations → ❌
Answer = 0.1:   Can we achieve max distance of 0.1? → Need too many stations → ❌
Answer = 0.5:   Can we achieve max distance of 0.5? → Need exactly 4 stations → ✅
Answer = 0.6:   Can we achieve max distance of 0.6? → Need 3 stations → ✅
Answer = 1.0:   Can we achieve max distance of 1.0? → Need 2 stations → ✅

Pattern: If answer X is possible, all answers > X are also possible
         We want the MINIMUM possible answer
         → Perfect for binary search!
```

## 📏 Determining the Search Range

### Lower Bound: Why `low = 0`?

```python
low = 0.0  # Minimum possible answer
```

**Reasoning:**

```
Scenario: All gas stations at the SAME coordinate

Gas stations: [1, 1, 1, 1]
k = 4 (new stations to place)

We can place all new stations at coordinate 1:
[1, 1, 1, 1, 1, 1, 1, 1]

Distance between any two consecutive = 0

Therefore, minimum possible answer = 0
```

**Real example:**

```python
stations = [5, 5, 5]  # All at position 5
k = 3

# Place new stations also at 5
result = [5, 5, 5, 5, 5, 5]

# Max distance = 0
```

### Upper Bound: Why `high = max(consecutive_differences)`?

```python
# Find maximum distance in original array
high = 0.0
for i in range(1, n):
    diff = arr[i] - arr[i-1]
    high = max(high, diff)
```

**Reasoning:**

```
Gas stations: [1, 2, 4, 6, 12]
                ↓  ↓  ↓  ↓
Differences:    1  2  2  6  ← Maximum = 6

Current maximum distance = 6

Question: Can the answer be > 6?

Answer: NO! Why?
- We're trying to MINIMIZE the maximum distance
- Even if we place NO new stations, max distance = 6
- Placing stations can only REDUCE this, never increase
- Therefore, answer ≤ 6
```

**Visual proof:**

```
Before: [1, 2, 4, 6, 12]
         └─1─┘ └─2─┘ └─2─┘ └───6───┘
         
Max distance = 6

After placing k stations optimally:
[1, 2, 4, 6, 8, 10, 12]
 └─1─┘ └─2─┘ └─2─┘ └─2─┘ └─2─┘ └─2─┘
 
Max distance = 2 (reduced from 6)

Can it ever be > 6? NO!
```

### Complete Range Determination Code

```python
def find_search_range(arr):
    """
    Determine the range [low, high] for binary search
    
    Args:
        arr: Array of gas station positions (sorted)
    
    Returns:
        tuple: (low, high) as floats
    """
    n = len(arr)
    
    # Lower bound: theoretical minimum
    low = 0.0
    
    # Upper bound: current maximum distance
    high = 0.0
    for i in range(1, n):
        consecutive_diff = arr[i] - arr[i-1]
        high = max(high, consecutive_diff)
    
    return low, high

# Example usage
stations = [1, 2, 4, 6, 12]
low, high = find_search_range(stations)
print(f"Search range: [{low}, {high}]")  # [0.0, 6.0]
```

## 🔍 The Check Function Logic

### What Does the Check Function Do?

> **Purpose**: Given a target maximum distance `dist`, determine if we can place `k` gas stations such that no two consecutive stations are more than `dist` apart.

### Core Question

```
Given: Target distance = 0.5
Question: How many gas stations do we NEED to maintain this distance?

If needed ≤ k → Possible ✅
If needed > k → Not possible ❌
```

---

### Calculating Required Stations for a Section

#### The Mathematical Formula

For a section from position `a` to position `b`:

```
Section length = b - a
Target distance = dist

Number of stations needed = ⌊(b - a) / dist⌋

Special case: If (b - a) is perfectly divisible by dist
              → Need one LESS station
```

#### Why This Formula Works

**Example 1: Section length = 1, Target distance = 0.4**

```
Positions: [1] ─────────── [2]
           └──── 1.0 ────┘

Target: Keep max distance ≤ 0.4

Calculation: 1.0 / 0.4 = 2.5 → ⌊2.5⌋ = 2

Place 2 stations:
[1] ── [1.4] ── [1.8] ── [2]
   0.4    0.4      0.2

✓ All distances ≤ 0.4
```

**Example 2: Section length = 1, Target distance = 0.5**

```
Positions: [1] ─────────── [2]
           └──── 1.0 ────┘

Target: Keep max distance ≤ 0.5

Calculation: 1.0 / 0.5
