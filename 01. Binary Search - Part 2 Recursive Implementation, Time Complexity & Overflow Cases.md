## 🔄 Recursive Binary Search Deep Dive

### What is Recursive Binary Search?

> **Recursion** means a function calling itself with modified parameters until it reaches a base case (stopping condition).

In recursive binary search, instead of using a loop, the function calls itself with updated `low` and `high` pointers.

### Why Use Recursion?

**Advantages:**
- More **elegant and readable** code
- Natural fit for **divide-and-conquer** algorithms
- Easier to understand the **logic flow**

**Disadvantages:**
- Uses **call stack memory** (space complexity O(log n))
- Slightly **slower** than iterative due to function call overhead
- Risk of **stack overflow** for very large inputs

### When to Use Recursive vs Iterative?

| Scenario | Preferred Approach | Reason |
|----------|-------------------|--------|
| Production code | Iterative | Better space complexity O(1) |
| Interviews | Either | Shows versatility |
| Educational purposes | Recursive | Clearer logic demonstration |
| Very large datasets | Iterative | Avoids stack overflow |
| Tree/Graph problems | Recursive | Natural fit for hierarchical data |

**Why low > high?**
- We've exhausted the search space
- The target doesn't exist in the array
- This is the **termination condition**

### Return Value Propagation

```
Call 4 returns -1 → Call 3 returns -1 → Call 2 returns -1 → Call 1 returns -1
```

Each recursive call receives `-1` from its child call and propagates it upward until the original caller receives the final result.

## 💻 Complete Code Implementation

### Recursive Binary Search (Full Implementation)

```python
def binarySearch(nums, low, high, target):
    """
    Recursive binary search implementation
    
    Args:
        nums: Sorted array of integers
        low: Starting index of search space
        high: Ending index of search space
        target: Element to search for
    
    Returns:
        Index of target if found, -1 otherwise
    
    Time Complexity: O(log n)
    Space Complexity: O(log n) due to recursion stack
    """
    
    # BASE CASE: Search space is exhausted
    # When low > high, element doesn't exist
    if low > high:
        return -1
    
    # STEP 1: Calculate middle index
    # Using (low + high) // 2 for now
    # We'll discuss overflow-safe version later
    mid = (low + high) // 2
    
    # STEP 2: Check if middle element is the target
    if nums[mid] == target:
        return mid  # Found! Return the index
    
    # STEP 3: Target is in the right half
    # All elements from low to mid are smaller than target
    elif nums[mid] < target:
        # Recursively search right half
        # New search space: [mid+1, high]
        return binarySearch(nums, mid + 1, high, target)
    
    # STEP 4: Target is in the left half
    # All elements from mid to high are larger than target
    else:  # nums[mid] > target
        # Recursively search left half
        # New search space: [low, mid-1]
        return binarySearch(nums, low, mid - 1, target)


# Wrapper function for LeetCode-style problems
def search(nums, target):
    """
    Main function that initiates binary search
    
    Args:
        nums: Sorted array
        target: Element to find
    
    Returns:
        Index of target or -1
    """
    # Initial call with full array range
    # low starts at 0 (first index)
    # high starts at len(nums)-1 (last index)
    return binarySearch(nums, 0, len(nums) - 1, target)


# Example usage
if __name__ == "__main__":
    # Test case 1: Element exists
    arr1 = [3, 4, 6, 7, 9, 12, 16, 17]
    target1 = 12

    # Test case 3: Single element array

    target3 = 5

    # Test case 4: Empty array
    arr4 = []
    target4 = 1
    result4 = search(arr4, target4)
    print(f"Searching for {target4}: Index = {result4}")  # Output: -1

## ⏱️ Time Complexity Analysis

### Understanding Time Complexity

> **Time Complexity** measures how the number of operations grows as input size increases.

### Mathematical Derivation

#### Step 1: Understand the Pattern

Let's trace how search space reduces:

```
Initial size: n = 32
After 1 step: ~16  (n/2)
After 2 steps: ~8   (n/4)
After 3 steps: ~4   (n/8)
After 4 steps: ~2   (n/16)
After 5 steps: ~1   (n/32)
```

**Pattern:** Each step divides the search space by approximately 2.

#### Step 2: Why "Approximately"?

```python
# Example 1: Even split
low = 0, high = 7
mid = 3
# Left half: [0,1,2] = 3 elements
# Right half: [4,5,6,7] = 4 elements

# Example 2: Uneven split
low = 0, high = 6
mid = 3
# Left half: [0,1,2] = 3 elements
# Right half: [4,5,6] = 3 elements
```

The split isn't always exactly half, but it's **approximately n/2**, which is sufficient for complexity analysis.

#### Step 3: Count the Steps

For an array of size `n`, how many times can we divide by 2 until we reach 1?

```
n → n/2 → n/4 → n/8 → ... → 1
```

This is asking: **2^k = n**, solve for k.

**Answer:** k = log₂(n)

#### Step 4: Operations Per Step

At each step, we perform:
- 1 comparison (`nums[mid] == target`)
- 1 comparison (`nums[mid] < target`)
- 1 arithmetic operation (calculate `mid`)

All are **O(1)** constant time operations.

#### Step 5: Total Time Complexity

```
Total operations = (Operations per step) × (Number of steps)
                 = O(1) × O(log n)
                 = O(log n)
```

### Visual Proof with Example

**Array size: 32 elements**

```
Step 0: [32 elements] ────────────────────────────────────
                      │
Step 1: [16 elements] ────────────────────
                      │
Step 2: [8 elements]  ────────────
                      │
Step 3: [4 elements]  ──────
                      │
Step 4: [2 elements]  ───
                      │
Step 5: [1 element]   ─
```

**Total steps = 5**

**Mathematical verification:**
```
log₂(32) = log₂(2^5) = 5 × log₂(2) = 5 × 1 = 5 ✓
```
