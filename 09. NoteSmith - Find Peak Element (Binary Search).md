## 🎯 Core Concept: Peak Element

### What is a Peak Element?

> A **peak element** is an element that is strictly greater than its neighbors. For elements at the boundaries, we only compare with one neighbor.

### Key Properties
1. **No Equal Consecutive Elements**: The problem guarantees that `arr[i] != arr[i+1]` (no flat lines)
2. **Multiple Peaks Possible**: An array can have more than one peak
3. **Any Peak is Valid**: If multiple peaks exist, returning any one is acceptable
4. **Boundary Handling**: First and last elements only need to be greater than one neighbor

### Example Arrays

```python
# Single Peak
arr1 = [1, 2, 3, 4, 5, 3, 1]  # Peak at index 4 (value 5)

# Multiple Peaks
arr2 = [1, 3, 2, 5, 4, 7, 6]  # Peaks at indices 1, 3, 5 (values 3, 5, 7)

# Peak at Boundary
arr3 = [5, 4, 3, 2, 1]  # Peak at index 0 (value 5)
arr4 = [1, 2, 3, 4, 5]  # Peak at index 4 (value 5)
```

## 📊 Single Peak vs Multiple Peaks

### Single Peak Scenario

When visualized as a graph, a single-peak array looks like a mountain:

```
        *  (peak)
       / \
      /   \
     /     \
    /       \
   /         \
```

**Key Insight**: 
- If `mid` is on the **increasing slope** → peak is on the **right**
- If `mid` is on the **decreasing slope** → peak is on the **left**
- If `mid` is the **peak** → return it

## 📈 Visual Understanding with Graphs

### The Three Cases Analysis

Let's analyze each case with multiple peaks present:

#### Case 1: Mid is on a Peak
```
    *  ← mid here
   / \
  /   \
```
**Action**: Return immediately  
**Why it works**: You found a peak, job done!

#### Case 2: Mid is on Decreasing Slope
```
      *
     / \
    /   \ ← mid here
   /     \
  *       *  (other peaks exist on right)
```
**Action**: Move left (`high = mid - 1`)  
**Why it works**: Even though we're discarding the right half (which has peaks), the left half is **guaranteed** to have at least one peak because we're on a decreasing slope.

#### Case 3: Mid is on Increasing Slope
```
  *       *  (other peaks exist on left)
   \     /
    \   / ← mid here
     \ /
      *
```
**Action**: Move right (`low = mid + 1`)  
**Why it works**: Even though we're discarding the left half (which has peaks), the right half is **guaranteed** to have at least one peak because we're on an increasing slope.

### The Valley Case (Critical Edge Case!)

```
  *       *
   \     /
    \   / ← mid here (valley/trough)
     \ /
      V
```

**Problem**: Mid is neither on a peak, nor clearly on an increasing/decreasing slope—it's in a **valley** (reverse peak).

**Example**:
```python
arr = [5, 1, 2]
# low = 0, high = 2, mid = 1
# arr[mid] = 1
# arr[mid] < arr[mid-1] (1 < 5) ✓
# arr[mid] < arr[mid+1] (1 < 2) ✓
# Neither increasing nor decreasing case matches!
```

**Solution**: Go **either left OR right**—both directions are guaranteed to have a peak!

## 🔧 Algorithm Development

### Step-by-Step Logic

#### Step 1: Handle Boundary Cases First
Before binary search, check if peaks exist at boundaries:

```python
# If array has only 1 element, it's a peak
if n == 1:
    return 0

# If first element is greater than second, it's a peak
if arr[0] > arr[1]:
    return 0

# If last element is greater than second-to-last, it's a peak
if arr[n-1] > arr[n-2]:
    return n - 1
```

**Why handle these first?**
- Avoids index out of bounds errors
- Simplifies the main binary search logic
- These are O(1) checks that can save O(log n) work

#### Step 2: Binary Search on Remaining Elements

```python
low = 1          # Start from second element
high = n - 2     # End at second-to-last element
```

**Why trim the search space?**
- We already checked boundaries
- Now every element has both left and right neighbors
- No need for additional boundary checks inside the loop

#### Step 3: Three Conditions in Binary Search

```python
while low <= high:
    mid = (low + high) // 2
    
    # Condition 1: Mid is a peak
    if arr[mid] > arr[mid-1] and arr[mid] > arr[mid+1]:
        return mid
    
    # Condition 2: Mid is on increasing slope
    elif arr[mid] > arr[mid-1]:
        low = mid + 1  # Peak is on the right
    
    # Condition 3: Mid is on decreasing slope OR in a valley
    else:
        high = mid - 1  # Peak is on the left
```

**Why the `else` works for both decreasing slope and valley:**
- If decreasing: left definitely has a peak
- If valley: both sides have peaks, going left is valid

## 💻 Code Implementation

### Complete Solution with Line-by-Line Explanation

```python
def findPeakElement(arr):
    """
    Find a peak element in an array using Binary Search.
    
    Args:
        arr: List of integers with no consecutive equal elements
    
    Returns:
        Index of any peak element
    
    Time Complexity: O(log n)
    Space Complexity: O(1)
    """
    n = len(arr)
    
    # Edge Case 1: Single element array
    # A single element is always a peak (no neighbors to compare)
    if n == 1:
        return 0
    
    # Edge Case 2: First element is a peak
    # Only need to check if arr[0] > arr[1]
    if arr[0] > arr[1]:
        return 0
    
    # Edge Case 3: Last element is a peak
    # Only need to check if arr[n-1] > arr[n-2]
    if arr[n-1] > arr[n-2]:
        return n - 1
    
    # Binary Search on the remaining elements (indices 1 to n-2)
    # We've already checked boundaries, so every element here has two neighbors
    low = 1
    high = n - 2
    
    while low <= high:
        # Calculate middle index
        mid = (low + high) // 2
        
        # Case 1: Mid is a peak element
        # It's greater than both left and right neighbors
        if arr[mid] > arr[mid - 1] and arr[mid] > arr[mid + 1]:
            return mid
        
        # Case 2: Mid is on an increasing slope
        # arr[mid] > arr[mid-1] means we're going upward
        # The peak must be on the right side
        elif arr[mid] > arr[mid - 1]:
            low = mid + 1  # Eliminate left half
        
        # Case 3: Mid is on a decreasing slope OR in a valley
        # If arr[mid] < arr[mid-1], we're going downward
        # OR if arr[mid] < both neighbors (valley)
        # In both cases, moving left is guaranteed to find a peak
        else:
            high = mid - 1  # Eliminate right half
    
    # This line should never be reached if input is valid
    # Added for completeness to satisfy function return requirements
    return -1


# Test Cases
if __name__ == "__main__":
    # Test Case 1: Single peak in middle
    arr1 = [1, 2, 3, 4, 5, 3, 1]
    print(f"Peak in {arr1} is at index: {findPeakElement(arr1)}")  # Output: 4
    
    # Test Case 2: Multiple peaks
    arr2 = [1, 3, 2, 5, 4, 7, 6]
    print(f"Peak in {arr2} is at index: {findPeakElement(arr2)}")  # Output: 1, 3, or 5
    
    # Test Case 3: Peak at start
    arr3 = [5, 4, 3, 2, 1]
    print(f"Peak in {arr3} is at index: {findPeakElement(arr3)}")  # Output: 0
    
    # Test Case 4: Peak at end
    arr4 = [1, 2, 3, 4, 5]
    print(f"Peak in {arr4} is at index: {findPeakElement(arr4)}")  # Output: 4
    
    # Test Case 5: Valley case
    arr5 = [5, 1, 2]
    print(f"Peak in {arr5} is at index: {findPeakElement(arr5)}")  # Output: 0 or 2
    
    # Test Case 6: Single element
    arr6 = [42]
    print(f"Peak in {arr6} is at index: {findPeakElement(arr6)}")  # Output: 0
```

### Detailed Execution Trace

Let's trace through a valley case to understand the fix:

```python
arr = [5, 1, 2]
n = 3

# Boundary checks
if n == 1: False
if arr[0] > arr[1]: 5 > 1 → True
return 0  # Peak found at boundary!

low = 1, high = 1, mid = 1
arr[mid] = 1
arr[mid] > arr[mid-1]? 1 > 5? False
```

## ⚠️ Edge Cases & Common Mistakes

### Common Mistakes Junior Developers Make

#### Mistake 1: Not Handling Boundaries Separately
```python
# ❌ WRONG: This will cause index out of bounds
def findPeak(arr):
    low, high = 0, len(arr) - 1
    while low <= high:
        mid = (low + high) // 2
        # What if mid = 0? arr[mid-1] will be arr[-1]!
        if arr[mid] > arr[mid-1] and arr[mid] > arr[mid+1]:
            return mid
```

**Fix**: Always handle boundary cases first or add extra conditions.

#### Mistake 2: Forgetting the Valley Case
```python
# ❌ WRONG: Infinite loop possible
def findPeak(arr):
    # ... boundary checks ...
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] > arr[mid-1] and arr[mid] > arr[mid+1]:
            return mid
        elif arr[mid] > arr[mid-1]:
            low = mid + 1
        # Missing else case! If mid is in valley, no pointer moves!
```

**Fix**: Always have an `else` clause to handle the valley case.

#### Mistake 3: Using `mid` Instead of `mid ± 1`
```python
# ❌ WRONG: Can cause infinite loop
def findPeak(arr):
    # ... boundary checks ...
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] > arr[mid-1] and arr[mid] > arr[mid+1]:
            return mid
        elif arr[mid] > arr[mid-1]:
            low = mid  # Should be mid + 1
        else:
            high = mid  # Should be mid - 1
```

**Why it's wrong**: If `low = mid` and `mid` doesn't change, infinite loop occurs.

#### Mistake 4: Assuming Only One Peak Exists
```python
# ❌ WRONG THINKING: "I need to find THE peak"
# ✅ CORRECT THINKING: "I need to find A peak (any one is fine)"
```

This misconception leads to overthinking the problem.

### Production Edge Cases

#### Edge Case 1: Empty Array

```python
def findPeakElement(arr):

    # ... rest of code
```

#### Edge Case 2: All Elements Increasing

```python
arr = [1, 2, 3, 4, 5]
```

#### Edge Case 3: All Elements Decreasing
```python

# Boundary check catches this
```

#### Edge Case 4: Two Elements
```python
arr = [1, 2]
# If arr[0] > arr[1]: return 0
# Else: return 1 (last element check)
```

### Performance Pitfalls

#### Pitfall 1: Using Linear Search
```python
# ❌ O(n) solution - defeats the purpose
def findPeak(arr):
    for i in range(1, len(arr) - 1):
        if arr[i] > arr[i-1] and arr[i] > arr[i+1]:
            return i
```

**Why avoid**: Interviewers specifically want O(log n) solution.

#### Pitfall 2: Unnecessary Comparisons
```python
# ❌ Redundant checks
if arr[mid] > arr[mid-1] and arr[mid] > arr[mid+1]:
    return mid
elif arr[mid] < arr[mid-1] and arr[mid] < arr[mid+1]:  # Valley check
    # ... handle valley
elif arr[mid] > arr[mid-1]:  # Increasing
    low = mid + 1
```

**Better**: Use `else` for the valley
